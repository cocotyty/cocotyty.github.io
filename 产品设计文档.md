## 1. 产品概述

### 一句话
纯静态的 LLM 驱动叙事冒险游戏，用户自配 API，手机可玩，托管在 GitHub Pages。

### 核心体验
- 每次新游戏 = 掷骰生成一个独一无二的世界
- 玩家通过选择 + 自由输入推进故事
- LLM 作为游戏主持人，遵循节奏控制 + 桥段指引
- 世界观逐层揭露，3 幕 7 章完整叙事弧

### 目标用户
- LLM 玩家（有 API key，喜欢探索 AI 叙事）
- 小说爱好者（想参与故事创作）
- 手机端为主，桌面为辅

## 2. 技术架构

### 约束
- 纯静态，无后端，零构建
- 可部署到 GitHub Pages
- 所有数据存浏览器 localStorage
- LLM API 由前端直接调用（用户自配 endpoint + key）

### 技术栈
| 项目 | 选择 | 理由 |
|------|------|------|
| 语言 | 原生 JS (ES2022) | 零依赖 |
| 样式 | CSS + CSS Variables | 轻量 |
| 星空背景 | Canvas 2D | 性能好 |
| Markdown 渲染 | marked.js (CDN) | 轻量 |
| 图标 | Unicode / SVG | 零依赖 |
| PWA | manifest.json + sw.js | 可安装到手机桌面 |

### 状态详解

#### SPLASH (启动页)
- 显示游戏标题 + 无尽星空背景
- 三个按钮: [新游戏] [继续] [设置]
- [继续] 灰色不可用 (当无存档时)
- 自动检测 localStorage 中的存档

#### SETTINGS (设置页)
- API 端点 URL (输入框)
- API Key (密码框)
- 模型选择 (下拉: gpt-4o / claude-3-opus / claude-3-sonnet / custom)
- 调试模式开关 (ON: 跳过掷骰, 加载预设 World Bible)
- 温度滑块 (0.5 - 1.5, 默认 0.9)
- [保存] 按钮 → 存入 localStorage 'ss_config'

#### WORLD_GEN (掷骰定世)
- 4 层骰子，每层显示为「牌面」
- 每张牌正面 = 骰子结果，背面 = 骰面列表
- 点击牌面 → 该维度重新掷骰
- 点击锁定图标 → 锁定该维度 (不再重掷)
- 全部锁定 → [确认世界] 按钮亮起
- 动画: 掷骰时牌面旋转

#### WORLD_CONFIRM (确认世界)
- 展示完整 World Bible (人类可读)
- 显示骰子结果汇总表
- [命运开始] 按钮 → 进入 LLM 序幕生成

#### PROLOGUE (序幕生成)
- "命运的齿轮开始转动..." 加载动画
- 调用 LLM 生成开场故事 (不含玩家选择)
- 打字机效果显示
- 完成后自动进入主游戏循环

#### PLAY (主游戏循环)
- **核心循环**:
  1. 显示故事文本 (打字机效果)
  2. 底部出现 3 个选项按钮 + 一个自由输入框
  3. 玩家点击选项/输入文本 → 按钮变为"思考中..."
  4. 构建 LLM Prompt (包含节奏控制 + 上下文)
  5. 调用 LLM
  6. 解析回复 (提取 narrative + choices + state_changes)
  7. 保存到 localStorage
  8. 事件引擎判定 (是否触发随机事件)
  9. 如果事件触发，插入事件叙事
  10. 节拍控制器检查 (当前 beat 是否完成？完成则推进)
  11. 章节完成检查 (4-5 beats 完成？→ 章节结束)
  12. 重新显示故事 + 选项

### 5.3 World Bible XML Schema

参见 `示例设定_星穹编年史.md` 第玖节。XML 包含:
- meta / meta_rules / cosmic_setting / protagonist / initial_situation
- villain_arc (含5章时间线)
- factions (≥3个势力)
- core_npcs (≥3个NPC，含三层人格+象限+好感度)
- trope_deck (4张桥段牌)
- sandbox_modules (≥2个箱庭)
- power_levels (境界+标签+状态)
- story_state (初始状态)

### 5.4 故事大纲 XML Schema

```xml
<story_outline>
  <act number="1" name="失忆者的漂流" expected_turns="15-20">
    <chapter number="1" name="铁棺苏醒" theme="冷寂苏醒" turns="5-8" beat_count="4">
      <beat number="1" type="introduce" intent="..." />
      <beat number="2" type="build" intent="..." />
      <beat number="3" type="climax" intent="..." />
      <beat number="4" type="resolve" intent="..." />
    </chapter>
    <!-- 更多 chapter -->
  </act>
  <!-- 更多 act -->
</story_outline>
```

Beat 类型轮换规则:
- 4-beat 章: introduce → build → climax → resolve
- 5-beat 章: introduce → build → twist → climax → resolve

## 6. UI 布局设计

### 6.1 设计原则
- **Mobile-first**: 最小宽度 320px，最大 480px（手机竖屏）
- **桌面适配**: 居中容器，最大宽度 600px
- **深色主题**: 星空间黑底，文字白/青/金三色

## 7. 节奏控制系统

### 7.1 三层结构

```
游戏 (Game)         全生命周期
 └─ 卷 (Act)        3-5 卷, 每卷 15-20 轮
     └─ 章 (Chapter)  每卷 3-5 章, 每章 5-8 轮
         └─ 节拍 (Beat) 每章 4-5 节拍, 每节拍 1-3 轮

全游戏 ≈ 50-80 轮玩家选择
```

### 7.2 节拍类型

| 类型 | 英文 | 叙事功能 | 轮数 |
|------|------|----------|------|
| 引入 | introduce | 建立场景基调，展示环境 | 1-2 |
| 展开 | build | 深化冲突，收集信息 | 2-3 |
| 转折 | twist | 意外反转，引入新变量 | 1-2 |
| 高潮 | climax | 冲突爆发，节奏最高点 | 1-2 |
| 收束 | resolve | 问题解决，过渡到下一场景 | 1 |

### 7.3 轮流转中的节奏控制

每轮 LLM 调用时，prompt 中包含一个「节奏仪表盘」：

```xml
<pacing_dashboard>
  <outline_progress>
    <act>1/3 — 失忆者的漂流</act>
    <chapter>1/7 — 铁棺苏醒</chapter>
    <beat>2/4 — 展开 (build)</beat>
    <turns_this_chapter>3</turns_this_chapter>
    <expected_turns_range>5-8</expected_turns_range>
  </outline_progress>
  <beat_instruction>
    当前节拍目标: 展示空间站的诡异氛围，让主角发现第一个异常
    已完成: 主角从逃逸舱进入空间站，发现空无一人的食堂
    下一节拍: 危机降临 (空间站自毁/虚境显现)
    限制: 不要在此节拍揭示核心真相，保持悬疑
  </beat_instruction>
</pacing_dashboard>
```

### 7.4 动态调整

- 如果玩家在某 beat 花太多轮（超过预期 max），自动压缩后续 beat
- 如果玩家推进太快（选择直冲主线），插入场景描写/环境叙事来填充
- 每章结束时，LLM 生成一段「章末总结」，更新大纲进度

  │                      │──────────────────────>│                        │
  │                      │                       │                        │
  │                      │                       │  ⑥ 生成叙事 + 选项    │
  │                      │<──────────────────────│                        │

  │                      │    - 提取 narrative    │                        │

  │                      │    - 提取 state_changes │                       │
  │                      │                       │                        │

  │                      │                       │                        │
  │                      │  ⑪ 保存到 localStorage │                       │
  │                      │────────────────────────────────────────────────>│
  │                      │                       │                        │
  │                      │  ⑫ 显示故事 (打字机)   │                        │
  │<─────────────────────│                       │                        │
  │                      │                       │                        │
  │  ⑬ 看到新故事 + 选项  │                       │                        │

### 8.2 初始化流程

```

## 9. LLM Prompt 模板

### 9.1 序幕生成 Prompt

```xml
<system>
你是《星穹编年史》的游戏主持人。

用第三人称有限视角（跟随主角，不写他人内心）写出开篇故事。
主角当前失忆，性格底色冷静。
世界基调是无尽星空的浪漫与神秘。
语言有文学感，精炼不啰嗦，善用感官描写。
</system>

<world_bible>
${worldBibleXml}
</world_bible>

<task>
请根据以上世界设定，写出游戏的序幕（第1章第1节拍·引入）。

要求:
- 从主角在逃逸舱中苏醒开始写
- 展示失忆状态、环境氛围、紧迫感 (维生倒计时/被追杀)
- 约 3-5 段
- 用中文，第三人称"他"
- 不要在此揭示核心真相，只埋线索

输出格式:
```xml
<narrative>
故事正文（markdown格式）
</narrative>
```
</task>
```

### 9.2 主循环 Prompt

```xml
<system>
你是《星穹编年史》的游戏主持人。

规则:
1. 用第三人称有限视角续写故事
2. 根据玩家的选择推进剧情
3. 每次给出约 2-5 段的叙事
4. 保持角色性格一致（主角性格: ${protagonistPersonality}）
5. 语言有文学感，善用感官描写
6. 不要替主角做决定，不要写主角的内心结论
</system>

<world_bible_summary>
${worldBibleSummary}
</world_bible_summary>

<pacing_dashboard>
  <act>${currentAct}</act>
  <chapter>${currentChapter}</chapter>
  <beat type="${currentBeatType}" number="${currentBeatNumber}/${totalBeats}">
    ${beatInstruction}
  </beat>
  <chapter_progress>
    本章已进行 ${turnsThisChapter} 轮，预期 ${expectedTurnsMin}-${expectedTurnsMax} 轮
  </chapter_progress>
</pacing_dashboard>

<trope_hint>
本局可用桥段: ${availableTropes}
当前场景适合触发的桥段: ${suggestedTrope}
</trope_hint>

<recent_history>
${lastFewTurns}
</recent_history>

<player_action>
玩家选择了: ${playerChoice}
</player_action>

<task>
续写故事。然后提供 3 个不同的选项让玩家选择下一步行动。

输出格式:
```xml
<response>
<narrative>
（故事正文，markdown格式）
</narrative>
<choices>
  <choice id="a">选项A的文字</choice>
  <choice id="b">选项B的文字</choice>
  <choice id="c">选项C的文字</choice>
</choices>
</response>
```
</task>
```
